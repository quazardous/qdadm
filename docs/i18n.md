# i18n

Schema-derived translation keys, pluggable providers, signal-driven locale switching.

## Why

Make qdadm i18n-ready without adding weight to projects that don't translate. Skipping i18n is **free**; adopting it is **easy**.

| CREDO principle | i18n application |
|---|---|
| Declaration over implementation | Keys derive from the declarative schema; you don't write `t('...')` in field configs |
| Invisible storage | Translations come from pluggable providers — pages don't know they exist |
| Module boundaries | Each module declares its messages alongside its entities |
| Signals over imports | Locale change goes through the signal bus |
| Smart filters (source, not values) | Labels are computed from schema structure |

## Core thesis

**The entity/field schema IS the i18n key source.** Surveyed libraries (react-admin, Refine, AdminJS, AdminForth) require hand-written keys. qdadm derives them from module/entity/field names; providers only supply translations.

## Quick start

### 1. Declare messages from a module

```js
class BooksModule extends Module {
  async connect(ctx) {
    ctx.messages('en', {
      entities: {
        books: {
          label: 'Book',
          labelPlural: 'Books',
          fields: {
            title: 'Title',
            author: 'Author',
            genre: {
              _label: 'Genre',
              options: { fiction: 'Fiction', 'non-fiction': 'Non-Fiction' },
            },
          },
        },
      },
      nav: { sections: { Library: 'Library' }, routes: { 'book-stats': 'Stats' } },
    })

    ctx.messages('fr', {
      entities: {
        books: {
          label: 'Livre',
          labelPlural: 'Livres',
          fields: {
            title: 'Titre',
            author: 'Auteur',
            genre: { _label: 'Genre', options: { fiction: 'Fiction', 'non-fiction': 'Documentaire' } },
          },
        },
      },
      nav: { sections: { Library: 'Bibliothèque' }, routes: { 'book-stats': 'Statistiques' } },
    })

    ctx.entity('books', new EntityManager({
      name: 'books',
      fields: { title: { type: 'text' }, author: { type: 'text' }, genre: { type: 'select', /* … */ } },
    }))
  }
}
```

Notice no `label:` is needed in the field configs. Convention keys are derived from `entities.{entity}.fields.{name}`.

### 2. Switch locale

Public API is the signal bus:

```js
const signals = inject('qdadmSignals')
await signals.emit('locale:change', 'fr')
// → kernel loads bundles for 'fr' from each provider, merges them
// → emits 'locale:changed', 'fr'
// → wired composables relabel fields, columns, groups and nav items reactively
```

The demo ships a `LocaleSwitcher` component in the sidebar that does exactly this.

### 3. Translate from custom code

```vue
<script setup>
import { useI18n } from '@quazardous/qdadm'
const { t, locale } = useI18n()
</script>

<template>
  <h1>{{ t('entities.books.label') }}</h1>
  <p>{{ t('core.errors.tooShort', { min: 3 }) }}</p>
  <p>Current locale: {{ locale }}</p>
</template>
```

Outside a kernel, `useI18n()` returns a no-op shim (`t(key) === key`) so components remain usable in isolated tests.

## Key conventions

```
entities.{entity}.label
entities.{entity}.labelPlural
entities.{entity}.fields.{field}
entities.{entity}.fields.{field}.placeholder
entities.{entity}.fields.{field}.options.{value}
entities.{entity}.groups.{group}
entities.{entity}.actions.{action}
entities.{entity}.errors.{error}

modules.{module}.*           # module-scoped overrides (used by the 'module' strategy)

nav.sections.{section}
nav.routes.{route}

core.actions.{save|cancel|delete|edit|create|update|…}
core.fields.{id|created_at|updated_at|…}
core.messages.{empty|loading|noResults|noMatching|…}
core.tooltips.{saveAndContinue|saveAndReturn|unsavedChanges|…}
core.placeholders.{search|noSelection|…}
core.errors.{required|tooShort|tooLong|invalid|…}
```

You never write these keys in field configs. They appear only on the left-hand side of a `ctx.messages()` bundle, mirroring this shape.

The `_label` shorthand lets a node carry its own translation alongside children (used for select fields where the field has both a label and option labels).

## Fallback chain

```
1. lookup(currentLocale, key)
2. lookup(fallbackLocale, key)             ← defaults to defaultLocale
3. apply value-level @-alias (recursive, cycle-safe, '@@' escapes literal @)
4. apply pattern alias (longest-match wins, captures via $1, $2, …)
   → restart from step 1 with the rewritten key
5. for keys matching `entities.*.fields.{name}` only:
   → snakeCaseToTitle(field)               ← prototyping safety net
6. emit `i18n:missing`, return the raw key
```

Step 5 is intentionally narrow. `core.actions.save` missing must not silently humanise to "Save" — it surfaces as a real miss so you notice.

## Aliases

### Value-level (`@`)

A bundle leaf starting with `@` is a pointer to another key:

```js
ctx.messages('fr', {
  entities: {
    books: { actions: { save: '@core.actions.save' } },  // resolves to 'Enregistrer'
  },
})
```

Chains follow recursively, with cycle detection (depth cap 8). Use `@@` to escape a literal `@` (rare in practice).

### Pattern aliases

Wildcards with positional captures, declared per locale or via a strategy:

```ts
{ pattern: 'entities.*.actions.*', target: 'core.actions.$2' }
```

```js
ctx.aliases([
  { pattern: 'entities.*.fields.email', target: 'core.fields.email' },
])
```

Longest-match wins when patterns overlap.

## Key strategies

A strategy is a named preset of pattern aliases. Built-ins:

- **`global`** (default) — common concepts (CRUD actions, timestamp fields, validation errors) route to `core.*`. Maximum DRY.
- **`module`** — routes `entities.{entity}.actions.*` and `.errors.*` to `modules.{module}.*` (entity-to-module mapping is captured by `ctx.entity()` automatically).
- **`entity`** — empty preset. Every entity owns its labels.

Choose at three levels:

```js
new Kernel({ i18n: { keyStrategy: 'global' } })          // app default
ctx.i18n.strategy('module')                              // module-scoped (when needed)
```

Or compose:

```js
new Kernel({ i18n: { keyStrategy: ['global', 'ecommerce'] } })
```

Define your own:

```js
import { defineStrategy } from '@quazardous/qdadm'
defineStrategy('ecommerce', [
  { pattern: 'entities.*.fields.price', target: 'shop.fields.price' },
])
```

## Providers

The kernel owns translation backends through the `TranslationProvider` interface:

```ts
interface TranslationProvider {
  readonly name: string
  load(locale: string): Promise<MessagesBundle> | MessagesBundle
  availableLocales?(): string[] | Promise<string[]>
  save?(locale: string, bundle: MessagesBundle): Promise<void>
  watch?(locale: string, cb: (b: MessagesBundle) => void): () => void
}
```

Built-in: **`InlineTranslationProvider`** backs `ctx.messages()`. It is always present.

Register additional providers per module or at kernel boot:

```js
ctx.messagesProvider(new MyHttpProvider({ endpoint: '/api/i18n/{locale}' }))

new Kernel({
  i18n: {
    defaultLocale: 'en',
    fallbackLocale: 'en',
    providers: [new MyJsonProvider({ path: './locales/{locale}.json' })],
  },
})
```

The kernel calls `load()` for the default + fallback locales during warmup, and lazily loads other locales on `locale:change`.

External provider packages (JSON, YAML, HTTP, Tolgee, Paraglide) are out of scope of the core; structure is in place to add them as separate `@qdadm/translation-provider-*` packages without core refactor.

## Wired composables

These resolve labels through the convention keys automatically — you only need to declare messages.

| Composable | What it resolves |
|---|---|
| `useFieldManager` | `entities.{entity}.fields.{name}`, `entities.{entity}.groups.{name}` |
| `useEntityItemFormPage` / `useEntityItemShowPage` | Pass entity to `useFieldManager`; field labels follow the convention |
| `useListPage` | Column headers via `entities.{entity}.fields.{field}` |
| `useNavigation` | `nav.sections.{title}`, `nav.routes.{route}` |

All four watch the active locale and re-resolve labels on `locale:changed`.

`FormActions.vue` and `LookupPickerDialog.vue` resolve their UI strings through `core.actions.*` / `core.placeholders.*` / `core.messages.*`.

## Default core bundle

qdadm ships an English `core.*` bundle covering actions, fields, messages, tooltips, placeholders and errors. Override per project:

```js
ctx.messages('en', {
  core: {
    actions: { save: 'Save changes' }, // overrides shipped 'Save'
  },
})
```

Disable the default if you want a fully clean slate:

```js
new Kernel({ i18n: { disableDefaultCoreBundle: true } })
```

## Dev tools

- **`i18n:missing`** signal fires on every miss (steps 5 & 6 of the fallback chain). Hook a collector in dev to find untranslated keys:

  ```js
  signals.on('i18n:missing', ({ data }) => console.warn('[i18n missing]', data))
  ```

  In an untranslated app this fires on essentially every key and floods the debug panel's "Missing" section. Silence it with `emitMissing: false`:

  ```js
  new Kernel({ i18n: { emitMissing: false } })
  ```

  No functional impact — only the debug collector consumes this signal, so the "Missing" section just stays empty.

- **`kernel.i18n.resolve(key)`** returns the full resolution trace (lookup steps, alias chain, hit/miss). Useful when a label resolves to something unexpected.

- **`kernel.i18n.dump(locale)`** exports the merged inline bundle, suitable for handing to a translator.

- **`window.__qdadm.i18n`** is exposed in debug mode for browser-console inspection.

## Caveats

1. **PrimeVue locale**: DataTable, Calendar etc. keep their own English defaults. A `qdadm-locale → primevue-locale` mapping is on the deferred list. For now, configure PrimeVue locale separately if needed.
2. **Pluralization (ICU)**: messages support `{placeholder}` interpolation only. Plural/select forms are not handled by the built-in resolver. An optional `@intlify/message-compiler`-backed provider is on the deferred list.
3. **Project templates**: custom Vue templates in your modules (e.g. `<h1>Books Library</h1>` in `BookForm.vue`) are *your* code — not auto-translated. Use `useI18n()` and your own keys.
4. **Custom validators returning strings**: `validator: (v) => 'Too short'` won't auto-translate. Use `t('core.errors.tooShort')` inside the validator.
5. **Backend error messages**: out of scope. Map them to core keys in your error handler if you want them translated.
6. **Editor placeholders** (RoleForm, KeyValueEditor, ScopeEditor, …): some specialised editors still hold hardcoded `placeholder=` strings. They're rarely visible to end users; pull-request welcome.
